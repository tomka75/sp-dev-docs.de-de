## <a name="background"></a><span data-ttu-id="aa5ed-108">Hintergrund</span><span class="sxs-lookup"><span data-stu-id="aa5ed-108">Background</span></span>

<span data-ttu-id="aa5ed-p102">SharePoint wird schon lange als Anwendungs- und/oder Entwicklungsplattform verwendet und bietet zahlreiche Sätze von Entwicklungs- und Anpassungsoptionen – von voll vertrauenswürdiger Codeausführung auf den SharePoint-Servern selbst zu Sandkastenlösungen und Add-Ins bis hin zu Benutzeroberflächenanpassungen mit direkt verfügbaren Features oder mit Einbettung von JavaScript/CSS. Im mehrinstanzenfähigen SharePoint Online wurde voll vertrauenswürdiger Code nie unterstützt, und der Sandkasten-Codedienst ist veraltet. Die gängigsten Muster für Anpassungen von SharePoint Online waren Add-Ins, Remotecodeausführung (Codeausführung an anderer Stelle, z. B. in Azure) über die Standard-APIs und JavaScript-Einbettung. Obwohl die JavaScript-Einbettung eine sehr leistungsfähige Möglichkeit zum Erweitern von SharePoint war, erwies es sich als schwierig, mit dem beliebten Modell von SharePoint Online Schritt zu halten. SharePoint Framework soll diese Probleme lösen, indem es ein standardisiertes Framework für das Erstellen benutzerdefinierter Benutzeroberflächenerweiterungen sowie zum unterstützten und zukunftsfähigen Erstellen von Anwendungen auf SharePoint Online bereitstellt.</span><span class="sxs-lookup"><span data-stu-id="aa5ed-p102">SharePoint has been used as an application and/or development platform for a long time and offered numerous sets of development and customization options ranging from full-trust code executing on the SharePoint servers itself, to Sandboxed solutions, to Add-ins, to interface customizations achieved using out of the box features or JavaScript/CSS embedding. Within multi-tenant SharePoint Online, full-trust code has never been supported, and the Sandboxed code service has been deprecated. The most common patterns for customizations of SharePoint Online has been either through Add-ins, remote-code execution (code executing elsewhere, such as in Azure) through the standard API's, and JavaScript embedding. Although JavaScript embedding has been a very powerful way of extending SharePoint, it has also proven difficult of keeping up with the evergreen model of SharePoint Online. <span data-ttu-id="aa5ed-p103">Der Schwerpunkt von SharePoint Framework liegt anfänglich auf der Erweiterung der SharePoint-Benutzeroberfläche mit *clientseitigen Webparts*. SharePoint Framework verfügt außerdem über eine vollständig neue Gruppe von Tools. Diese neuen Tools sind plattformunabhängig, funktionieren auf PC und Mac und basieren auf Open-Source-Technologien wie [Node.js](https://nodejs.org/), [Gulp](http://gulpjs.com/), [Webpack](http://webpack.github.io/) und [Yeoman](http://yeoman.io/). Diese Frameworks und Tools werden zur Erstellungszeit verwendet, um die Entwicklerumgebung zum Erstellen, Packen und Bereitstellen zu optimieren. These client side Web Parts works on the new Modern pages as well as in classic pages and even in the SharePoint mobile app. This new model is built from ground-up using a modern web stack using TypeScript/JavaScript, HTML and CSS, and all parts of the generated artifacts are executed in the end-users browser. The SharePoint Framework also comes with a completely new set of tooling. This new tooling is platform agnostic and works on PC and Mac and is based on open source technologies such as [Node.js](https://nodejs.org/), [Gulp](http://gulpjs.com/), [Webpack](http://webpack.github.io/) and [Yeoman](http://yeoman.io/). ### <a name="compared-with-add-ins"></a><span data-ttu-id="aa5ed-144">Vergleich mit Add-Ins</span><span class="sxs-lookup"><span data-stu-id="aa5ed-144">Compared with Add-ins</span></span>

<span data-ttu-id="aa5ed-p109">SharePoint-Add-Ins, früher bekannt als SharePoint-Apps und eingeführt in SharePoint 2013, waren eine der Optionen zum unterstützten und gesteuerten Hinzufügen von Anpassungen zu SharePoint Online. SharePoint-Add-Ins gibt es in zwei Versionen: von SharePoint gehostet und vom Anbieter gehostet. Von SharePoint gehostete Add-Ins waren eine der Möglichkeiten zur unterstützten Ausführung von clientseitigem Code in SharePoint, erfordern jedoch wie gesagt wesentlich mehr Aufwand als nötig, um ein einfaches clientseitiges (JavaScript-)Webpart einzufügen. Diese Add-Ins befinden sich im so genannten App-Web, einer speziellen Website mit begrenzten Features. Vom Anbieter gehostete Add-Ins werden dagegen remote von SharePoint (Online) ausgeführt und können sowohl serverseitigen Code als auch clientseitigen Code nutzen. Dies hat einen Vorteil für unabhängige Softwarehersteller, die ihr geistiges Eigentum/ihren Code/ihre Logik schützen möchten, und für Szenarien, die nicht clientseitig mit JavaScript ausgeführt werden können. Der größte Vorteil von Add-Ins ist die Isolation: Da der eigentliche Code nicht auf der SharePoint-Website ausgeführt wird, verhindert der Schutz des Browsers vor websiteübergreifendem Skripting, dass das Add-In dieselben Zugriffsrechte wie der Benutzer erhält. Add-Ins sind auf die Berechtigungen beschränkt, die dem Add-In zur Installationszeit erteilt wurden. Dies macht Add-Ins zu einer sichereren Option für Szenarien, in denen ein Administrator ein Add-In eines Drittanbieters erwirbt, und es ermöglicht Microsoft die Einrichtung eines Stores, aus dem Sie Add-Ins herunterladen können.</span><span class="sxs-lookup"><span data-stu-id="aa5ed-p109">SharePoint Add-ins, previously known as SharePoint Apps and was introduced in SharePoint 2013, have been one of the only available options of adding customizations to SharePoint Online in a supported and governed way. SharePoint Add-ins comes in two flavors: SharePoint hosted and Provider hosted. SharePoint hosted Add-ins have been one of the ways of executing client side code in SharePoint in a supported way, but as said requires a lot more effort than necessary to just include a simple client side (JavaScript) Web Part. The primary advantage of Add-ins is isolation: since the actual code is not executed in the SharePoint site browser cross-site scripting protections prevent the Add-in from getting all of the same access as the user has. <span data-ttu-id="aa5ed-p110">SharePoint Framework funktioniert parallel mit von SharePoint gehosteten und vom Anbieter gehosteten Add-Ins, aber es kann auch als Alternative verwendet werden, wenn nur clientseitige Skripts erforderlich sind. <span data-ttu-id="aa5ed-p111">Für SharePoint Framework-Lösungen gibt es zum Zeitpunkt der Erstellung dieses Artikels aus den genannten Sicherheitsgründen keinen Store, aus dem Sie Lösungen herunterladen und installieren können.
