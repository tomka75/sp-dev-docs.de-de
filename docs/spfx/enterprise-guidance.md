# <a name="sharepoint-framework-spfx-enterprise-guidance"></a>SharePoint Framework (SPFx) Enterprise-Leitfaden

## <a name="introduction"></a>Einführung

SharePoint Framework (oder SPFx) ist ein neues Entwicklungsmodell, das die Erweiterung der SharePoint-Benutzeroberfläche ermöglicht.  Es wird von Erst- und Drittanbietern verwendet und ergänzt vorhandene Benutzeroberflächenmodelle, wie z. B. das SharePoint-Add-In-Modell. SharePoint Framework ermöglicht einen strukturierten und unterstützten Ansatz zur Erweiterung und Bereicherung der Benutzeroberfläche von SharePoint mithilfe clientseitiger Frameworks und Anfangsunterstützung für clientseitige Webparts. Basierend auf modernen Webtechnologiestandards, bietet es einen eindeutigen Satz von Features, um SharePoint-Anpassungen besser für Entwickler und Unternehmen verfügbar zu machen. Gleichzeitig ist es jedoch an früheren Modellen und Mustern in SharePoint ausgerichtet. In diesem Dokument bieten wir Administratoren den Hintergrund, die Vorteile und das Wissen, das sie benötigen, um SharePoint Framework-basierte Komponenten in ihren SharePoint-Umgebungen zu verwalten.

## <a name="background"></a>Hintergrund

SharePoint wird schon lange als Anwendungs- und/oder Entwicklungsplattform verwendet und bietet zahlreiche Sätze von Entwicklungs- und Anpassungsoptionen – von voll vertrauenswürdiger Codeausführung auf den SharePoint-Servern selbst zu Sandkastenlösungen und Add-Ins bis hin zu Benutzeroberflächenanpassungen mit direkt verfügbaren Features oder mit Einbettung von JavaScript/CSS. Im mehrinstanzenfähigen SharePoint Online wurde voll vertrauenswürdiger Code nie unterstützt, und der Sandkasten-Codedienst ist veraltet. Die gängigsten Muster für Anpassungen von SharePoint Online waren Add-Ins, Remotecodeausführung (Codeausführung an anderer Stelle, z. B. in Azure) über die Standard-APIs und JavaScript-Einbettung. Obwohl die JavaScript-Einbettung eine sehr leistungsfähige Möglichkeit zum Erweitern von SharePoint war, erwies es sich als schwierig, mit dem beliebten Modell von SharePoint Online Schritt zu halten. SharePoint Framework soll diese Probleme lösen, indem es ein standardisiertes Framework für das Erstellen benutzerdefinierter Benutzeroberflächenerweiterungen sowie zum unterstützten und zukunftsfähigen Erstellen von Anwendungen auf SharePoint Online bereitstellt.

Der Schwerpunkt von SharePoint Framework liegt anfänglich auf der Erweiterung der SharePoint-Benutzeroberfläche mit *clientseitigen Webparts*. Die clientseitigen Webparts verwenden das bekannte Paradigma von Webparts – im Lauf der Jahre einer der Erfolgsfaktoren von SharePoint – die zu Seiten hinzugefügt und unabhängig von den Endbenutzern angepasst werden können. Diese clientseitigen Webparts funktionieren auf den neuen modernen Seiten ebenso wie auf den klassischen Seiten und sogar in der mobilen SharePoint-App. Das neue Modell basiert vollständig auf einem modernen Webstapel mit TypeScript/JavaScript, HTML und CSS, und alle Teile der generierten Artefakte werden im Browser der Endbenutzer ausgeführt. SharePoint Framework verfügt außerdem über eine vollständig neue Gruppe von Tools. Diese neuen Tools sind plattformunabhängig, funktionieren auf PC und Mac und basieren auf Open-Source-Technologien wie [Node.js](https://nodejs.org/), [Gulp](http://gulpjs.com/), [Webpack](http://webpack.github.io/) und [Yeoman](http://yeoman.io/). Diese Frameworks und Tools werden zur Erstellungszeit verwendet, um die Entwicklerumgebung zum Erstellen, Packen und Bereitstellen zu optimieren. Für die tatsächliche Ausführung des SharePoint Framework-Codes sind sie nicht erforderlich.

## <a name="current-status-of-sharepoint-framework"></a>Aktueller Status von SharePoint Framework

SharePoint Framework erreicht im Februar 2017 mit der Version 1.0.0 den [Meilenstein der allgemeinen Verfügbarkeit (General Availability, GA)](https://blogs.office.com/2017/02/23/sharepoint-framework-reaches-general-availability-build-and-deploy-engaging-web-parts-today/).

## <a name="from-a-developers-point-of-view"></a>Standpunkt des Entwicklers

Neue wie auch erfahrene SharePoint-Entwickler profitieren von SharePoint Framework. Die aktuelle Version ist auf die Erweiterbarkeit der Benutzeroberfläche konzentriert und ermöglicht dem Entwickler die sichere und strukturierte Erweiterung der Benutzeroberflächenfunktionen von SharePoint mit anfänglich clientseitigen Webparts. Diese Webparts werden clientseitig ausgeführt und können Daten in SharePoint, in Office 365 über [Microsoft Graph](http://graph.microsoft.io) oder auch mithilfe Ihrer eigenen benutzerdefinierten Web-APIs mit standardmäßigen OAuth- und REST-Methoden verwenden.

Ein erfahrener SharePoint-Entwickler ist mit den Konzepten, wie z. B. Webparts und dem SharePoint-Datenmodell, vertraut. Die Tools zum Erstellen, Packen und Bereitstellen clientseitiger Webparts sind jedoch neu. Entwickler müssen insbesondere Kenntnisse in [TypeScript](http://www.typescriptlang.org/) erwerben, der primären Sprache für die Entwicklung von SharePoint Framework-Artefakten. TypeScript bietet mehrere Vorteile gegenüber JavaScript, die wichtig für die Unternehmensentwicklung sind, z. B. stark typisierte Objekte, Objektvererbung, Klassen und Schnittstellen, Konzepte, die allen derzeitigen .NET-, Java- und C/C++-Entwicklern vertraut sein sollten. Bezüglich der Erstellung und Verpackung stellt Visual Studio für Entwickler nicht mehr die einzige Option zum Schreiben von SharePoint-Lösungen dar. Dank der Verwendung von Open-Source-Technologien und Projekten wie node.js, [npm](https://www.npmjs.com/) und Gulp kann die Entwicklung in SharePoint Framework auf jeder beliebigen Plattform mit dem vom Entwickler bevorzugten Code-Editor oder IDE erfolgen, z. B. Visual Studio Code, Sublime oder sogar Editor.

Für Entwickler, die noch nie SharePoint-Lösungen erstellt haben, aber mit modernen Webtechnologien vertraut sind, sind die Hürden nicht allzu hoch, und viele Entwickler sind bereits auf die Clientseite der Entwicklung oder eine Kombination aus beiden gewechselt. Die clientseitige Entwicklung ermöglicht eine bessere, dynamischere und reaktionsfähigere Oberfläche für Benutzer und auch eine einfachere Umgebung für Entwickler. Aufgrund der Freiheit des Code-Editors und der Verwendung gängiger Open-Source-Frameworks und Technologien können viele Entwickler, die möglicherweise noch gar nicht innerhalb des Microsoft-Ökosystems gearbeitet haben, leicht den Einstieg in die Erstellung von SharePoint-Erweiterungen finden.

Eines der meistverwendeten Muster bei der SharePoint Online-Erweiterbarkeit war die JavaScript-Einbettung (auch als JavaScript-Einfügung bezeichnet). Bei dieser Methode verwenden Sie z. B. das Skript-Editor-Webpart, um beliebigen JavaScript-Code auf der Seite einzufügen, und fügen dann per DOM (Document Object Model)-Manipulation des Webbrowsers HTML, CSS und JavaScript ein, um eine Lösung oder Anwendung zu erstellen. Diese Methode hat viele Nachteile. Aufgrund ihrer Abhängigkeit vom Aufbau der HTML- und CSS-Struktur in SharePoint wurden sogar häufig Kunden daran gehindert, die neuen Features in SharePoint Online zu nutzen. SharePoint Framework ermöglicht eine bessere Möglichkeit, wenn auch noch keinen vollständigen Ersatz, für diese Anpassungen durch JavaScript-Einbettung. SharePoint Framework verwendet wie gesagt TypeScript, das einen relativ einfachen Übergang von JavaScript-Einbettungen zu einer standardisierten und zukunftssicheren Lösung ermöglicht. Die OfficeDev PnP-Initiative bietet ebenfalls Beispielprojekte und Richtlinien für diesen Übergang.

## <a name="in-perspective-sharepoint-framework-in-the-broader-sharepoint-platform"></a>Perspektive: SharePoint Framework in der umfassenderen SharePoint-Plattform

SharePoint Framework ist ein neues Modell, zusätzlich zu bereits vorhandenen Methoden, jedoch mit Fokus auf Mehrwert für Benutzeroberflächenanpassungen, z. B. clientseitige Webparts. Dieses Framework ist darauf ausgelegt, in Verbindung mit vorhandenen funktionierenden Modellen eingesetzt zu werden, aber auch die Erstellung neuer Benutzeroberflächenanpassungen in einer besser unterstützten und nachhaltigeren Weise zu erleichtern.

### <a name="compared-with-add-ins"></a>Vergleich mit Add-Ins

SharePoint-Add-Ins, früher bekannt als SharePoint-Apps und eingeführt in SharePoint 2013, waren eine der Optionen zum unterstützten und gesteuerten Hinzufügen von Anpassungen zu SharePoint Online. SharePoint-Add-Ins erfordern jedoch wesentlich mehr Infrastruktur, als in vielen Fällen nötig, in denen eine einfache Anpassung der Benutzeroberfläche erforderlich ist. SharePoint-Add-Ins gibt es in zwei Versionen: von SharePoint gehostet und vom Anbieter gehostet. Von SharePoint gehostete Add-Ins waren eine der Möglichkeiten zur unterstützten Ausführung von clientseitigem Code in SharePoint, erfordern jedoch wie gesagt wesentlich mehr Aufwand als nötig, um ein einfaches clientseitiges (JavaScript-)Webpart einzufügen. In vielen Fällen wurden in SharePoint gehostete Add-Ins erstellt, nur um Artefakte, z. B. Listen und Webparts, auf einer SharePoint-Website bereitzustellen. Diese Add-Ins befinden sich im so genannten App-Web, einer speziellen Website mit begrenzten Features. Vom Anbieter gehostete Add-Ins werden dagegen remote von SharePoint (Online) ausgeführt und können sowohl serverseitigen Code als auch clientseitigen Code nutzen. Dies hat einen Vorteil für unabhängige Softwarehersteller, die ihr geistiges Eigentum/ihren Code/ihre Logik schützen möchten, und für Szenarien, die nicht clientseitig mit JavaScript ausgeführt werden können. Dies gilt z. B. für lange ausgeführte, rechenintensive Vorgänge oder Zugriffe auf Remotedaten, die nicht mit clientseitigen Skripts ausgeführt werden können. Der größte Vorteil von Add-Ins ist die Isolation: Da der eigentliche Code nicht auf der SharePoint-Website ausgeführt wird, verhindert der Schutz des Browsers vor websiteübergreifendem Skripting, dass das Add-In dieselben Zugriffsrechte wie der Benutzer erhält. Add-Ins sind auf die Berechtigungen beschränkt, die dem Add-In zur Installationszeit erteilt wurden. Dies macht Add-Ins zu einer sichereren Option für Szenarien, in denen ein Administrator ein Add-In eines Drittanbieters erwirbt, und es ermöglicht Microsoft die Einrichtung eines Stores, aus dem Sie Add-Ins herunterladen können.

SharePoint Framework funktioniert parallel mit von SharePoint gehosteten und vom Anbieter gehosteten Add-Ins, aber es kann auch als Alternative verwendet werden, wenn nur clientseitige Skripts erforderlich sind. Beispielsweise können Add-Ins App-Webparts zu der Website hinzufügen, auf der sie gehostet werden. Diese App-Webparts ähneln Webparts, aber statt im Kontext der Seite werden sie in ihrer eigenen Domäne (App-Web oder vom Anbieter gehostetes Web) in einem Iframe auf der Seite ausgeführt. Dies verhindert, dass das Add-In den Benutzerkontext vom Rest der Seite übernimmt. SharePoint Framework wird dagegen nicht in einem Iframe ausgeführt. Auf diese Weise kann es nahtloser im Kontext der Seite ausgeführt werden, mit dem ganzen Leistungsumfang des Benutzers, der das Webpart anzeigt. Dies ist der Schlüssel für seine Ausführbarkeit mit umfassender Funktionalität, bedeutet aber auch, dass es nicht über das gleiche Maß an Sicherheitssteuerelementen wie Add-Ins verfügt. SharePoint Framework-Lösungen werden daher auch als *voll vertrauenswürdige clientseitige Lösungen* bezeichnet. Auch bei Iframes besteht das Problem, dass sie nicht reaktionsfähig sind, sodass die gerenderte Webseite auf einem Mobiltelefon oder bei einer alternativen Bildschirmgröße nicht so flüssig dargestellt wird.

Für SharePoint Framework-Lösungen gibt es zum Zeitpunkt der Erstellung dieses Artikels aus den genannten Sicherheitsgründen keinen Store, aus dem Sie Lösungen herunterladen und installieren können. Andererseits ist die Verwendung des Benutzerkontexts häufig ein gewünschtes Szenario, in dem stattdessen SharePoint Framework verwendet werden kann.

### <a name="javascript-embeddings"></a>JavaScript-Einbettungen

Eine der gängigen Vorgehensweisen von Entwicklern bestand in einer Methode namens JavaScript-Einbettungen (auch als JavaScript-Einfügungen bezeichnet). Das bedeutet, dass beliebiges JavaScript auf den Websites und Seiten eingefügt wurde, z. B. mithilfe von benutzerdefinierten Aktionen, Gestaltungsvorlagen oder Seitenlayouts und sogar Skript-Editor-Webparts. Diese Methode erwies sich als einfacher als das Erstellen von in SharePoint gehosteten Add-Ins und ermöglicht außerdem, den Skriptcode im vollständigen Kontext der Benutzer auszuführen. Daher wurde sie sehr populär. Der Nachteil dieses Ansatzes ist, dass viele dieser Einbettungen auf DOM-Manipulation beruhten und ihre Erstellung und Wartung Entwicklerwissen erforderte. Und aufgrund der andauernden Beliebtheit von SharePoint Online konnten diese mit JavaScript-Einbettungen erstellten Lösungen bei jedem Update von SharePoint Online unbrauchbar werden, da die Entwickler möglicherweise Abhängigkeiten (auch unbeabsichtigt) von der Struktur oder Formatierung der SharePoint-Seiten verwendeten. Wenn Updates in SharePoint ausgeführt werden, auch Nebenversionen und geringfügige Updates, kann dies große Auswirkungen auf diese Lösungen haben und dazu führen, dass das eingebettete JavaScript nicht mehr ausgeführt werden kann.
Mit SharePoint-Framework steht nun eine von Microsoft standardisierte und unterstützte Möglichkeit zur Verfügung, um viele dieser zuvor mit JavaScript-Einbettungen erstellten Lösungen zu realisieren.

### <a name="script-editor-web-parts"></a>Skript-Editor-Webparts

Das am häufigsten verwendete Verfahren zum Einfügen von beliebigen HTML-, JavaScript- oder CSS-Anpassungen in SharePoint ist das Skript-Editor-Webpart oder das Inhalts-Editor-Webpart. Skript-Editor-Webparts sind beliebt, weil es so einfach ist, benutzerdefinierte Skripts zu einer beliebigen Seite hinzuzufügen. Jeder Bearbeiter einer Website kann ein Skript-Editor-Webpart auf einer Seite hinzufügen, ein JavaScript kopieren und dort einfügen und dieses JavaScript die notwendigen Anpassungen ausführen lassen. Wie bei JavaScript-Einbettungen kann es eine Herausforderung für Administratoren darstellen, die Kontrolle über Skript-Editor-Webparts zu behalten. SharePoint Framework kann in vielen Fällen ein direkter Ersatz für diese Konfigurationen mit Skript-Editor-Webparts sein.

### <a name="control-of-scripting-capabilities-in-sharepoint-online"></a>Steuerung von Skriptfunktionen in SharePoint Online

SharePoint Online ermöglicht Administratoren die Steuerung der Möglichkeit, Websites und Seiten benutzerdefinierte Skripts hinzufügen, um die Sicherheit und Integrität des Mandanten zu erhöhen. Dies geschieht mithilfe des Features „Benutzerdefiniertes Skript“ auf der SharePoint Online-Verwaltungswebsite oder einzeln pro Website mithilfe der PowerShell. Benutzerdefinierte Skripts können auf allen Websites oder nur auf persönlichen Websites deaktiviert werden. Standardmäßig ist für neue Mandanten das Skripting auf persönlichen Websites, allen per Self-Service erstellten Websites und der Stammwebsitesammlung des Mandanten deaktiviert. Wenn benutzerdefinierte Skripts deaktiviert sind, können Bearbeiter von Websites keine Webparts wie das Skript-Editor-Webpart hinzufügen. Jedoch sind SharePoint Framework-Lösungen zulässig, da diese als sicher gelten, nachdem sie von einem Administrator im App-Katalog genehmigt wurden.

## <a name="differences-in-how-sharepoint-framework-solutions-are-created--and-why-it-matters"></a>Unterschiede bei der Erstellung von SharePoint Framework-Lösungen und warum diese eine Rolle spielen

SharePoint Framework liegt ein neues Paradigma für SharePoint-Entwickler beim Entwerfen, Erstellen und Bereitstellen von SharePoint-Anpassungen zugrunde, das einen modernen Webstapelansatz verwendet und auf clientseitige/browserbasierte Anpassungen konzentriert ist. Dies bedeutet eine wichtige Änderung in der Behandlung der SharePoint-Entwicklung. Durch die Verwendung von Technologien und Frameworks wie z. B. TypeScript, Node.js, Yeoman und Gulp zieht SharePoint Framework hoffentlich Entwicklerzielgruppen an, die bisher nicht im Ökosystem von SharePoint oder sogar Microsoft tätig waren. Aber gleichzeitig soll es bestehenden SharePoint-Entwicklern das Erstellen von SharePoint-Anpassungen mit einem modernen und standardisierten Ansatz erleichtern.

### <a name="creating-solutions"></a>Erstellen von Lösungen

Aufgrund der benötigten, sehr spezifischen, von Visual Studio bereitgestellten Tools erfolgte die SharePoint-Entwicklung in der Regel ausschließlich über Visual Studio auf einem Windows-Computer mit einer installierten und aktiven Instanz von SharePoint, was erheblich die Hardwareauswahl und Benutzereinstellungen beschränkte und die Entwicklungskosten erhöhte. SharePoint Framework verwendet dagegen verschiedene gängige Open-Source-Webtools, die für eine Reihe von Plattformen verfügbar sind, wie z. B. Mac OS und Linux, um größere Flexibilität bei der Entwicklung zu ermöglichen.

SharePoint Framework-Lösungen werden mit einem Tools namens [Yeoman](http://yeoman.io/) zusammen mit einem bestimmten SharePoint Framework-Generator erstellt, der auf Node.js basiert. Yeoman ist ein Tool zum Erstellen eines Projektgerüsts. Es erstellt das Projekt und generiert die erforderlichen Artefakte, installiert die benötigten Node.js-Pakete und konfiguriert das Buildsystem. Nachdem das Projekt generiert wurde, kann es in einem beliebigen Editor unter einem beliebigen Betriebssystem bearbeitet werden, z. B. Visual Studio, Visual Studio Code, Sublime oder Atom. Dadurch wird eine größere Vielfalt in Nutzungsverhalten und Stilen in und zwischen Teams ermöglicht. Die Yeoman-Generator kann mehrfach für dasselbe Projekt ausgeführt werden, um zusätzliche Elemente, z. B. clientseitige Webparts, hinzuzufügen.

### <a name="developing-and-building-solutions"></a>Entwickeln und Erstellen von Lösungen

Das Buildsystem basiert auf [Gulp](http://gulpjs.com/), einer Taskausführung, die Pakete erstellt und optional die SharePoint Framework-Artefakte bereitstellt. Wie Yeoman basiert auch Gulp auf Node.js und ermöglicht Entwicklern das Erstellen und Bereitstellen unter einem beliebigen Betriebssystem. Ein großer und neuer Teil des Buildtoolsets für SharePoint Framework wird als *Workbench* bezeichnet. Die Workbench ist ein Tool, mit dem der Entwickler seine SharePoint Framework-Lösung hosten und testen kann. Die Workbench ist reaktionsfähig und lädt die Artefakte automatisch neu, wenn der Entwickler eine Datei speichert, sodass Entwickler die Lösung schnell anzeigen und testen können. Es gibt zwei Version der Workbench. Eine wird außerhalb von SharePoint lokal auf dem Entwicklungscomputer gehostet, der „offline“ ohne SharePoint-Zugriff und SharePoint-Daten ausgeführt wird. Mit dieser können Team und Designer Lösungen mit simulierten oder falschen Daten erstellen und entwerfen, um sich auf die Benutzeroberfläche zu konzentrieren. Die zweite Version wird in SharePoint gehostet. Sie wird verwendet, wenn Sie die SharePoint Framework-Lösung mit realen SharePoint-Daten und dem Kontext testen und überprüfen müssen.

### <a name="deploying-sharepoint-framework-solutions"></a>Bereitstellen von SharePoint Framework-Lösungen

Das Bereitstellen von SharePoint Framework-Lösungen erfolgt in zwei Schritten. Im ersten werden die vom Buildprozess gepackten Skriptartefakte an einem CDN-Speicherort (Content Delivery Network) bereitgestellt. Im zweiten Schritt wird das Lösungspaket dem App-Katalog hinzugefügt und für die Verwendung in Ihrem Mandanten genehmigt. Das dem SharePoint App-Katalog hinzugefügte Paket enthält einen Verweis auf den CDN-Speicherort.

Entwickler einer SharePoint Framework-Lösung können jeden beliebigen CDN-Dienst verwenden, z. B. Azure Storage, Azure CDN oder auch SharePoint selbst, bevorzugt mithilfe der SharePoint CDN-Funktionen (siehe unten). Bei Verwendung eines öffentlichen CDN, wobei die im CDN bereitgestellten Objekte im Internet öffentlich verfügbar sind, kann die SharePoint Framework-Lösung von vielen Mandanten verwendet werden. Bei einer in SharePoint CDN bereitgestellten SharePoint Framework-Lösung sind die bereitgestellten Skripts und Ressourcen nur für den Mandanten verfügbar, in dem die Lösung bereitgestellt wird.

Standardmäßig enthalten die Buildtools die integrierte Aufgabe, die gepackte Lösung in einem Azure-Blobspeicher bereitzustellen. Dies wird in der Regel von Systemintegratoren oder ISVs erweitert, um benutzerdefinierte CDN-Speicherorte oder Konfigurationen zu unterstützen.

Nach Änderung des Codes und Erstellung der Lösung erzeugt die SharePoint-Framework-Toolkette ein neues Lösungspaket (.sppkg) und eine Reihe von Skriptdateien. Diese Skriptdateien enthalten einen eindeutigen Hash im Dateinamen, welcher angibt, dass der Inhalt dieser Dateien von den zuvor bereitgestellten Versionen abweicht. Um eine neue Version der Lösung zu verwenden, müssen Sie die neuen Skripts im CDN bereitstellen und das Lösungspaket im App-Katalog aktualisieren. Theoretisch könnten Sie den Inhalt der vorhandenen Skriptdateien ersetzen, statt das Lösungspaket zu aktualisieren, dies ist jedoch unzuverlässig und wird nicht empfohlen. Je nach Konfiguration des CDN werden möglicherweise zuvor heruntergeladene Skriptdateien für längere Zeit auf den Clientcomputern zwischengespeichert, was das Rollout der Lösung für die Endbenutzer erschwert.

Der Speicherort des CDN ist von Bedeutung. Der Speicherort, in dem die SharePoint Framework-Ressourcen gehostet werden, muss hohe Verfügbarkeit aufweisen. Daher werden vertrauenswürdige CDN-Anbieter wie Azure, Akamai oder ähnliche und SharePoint selbst empfohlen. Unter Sicherheitsgesichtspunkten ist es wichtig zu wissen, welche CDNs von den bereitgestellten SharePoint Framework-Lösungen verwendet werden. Ein fehlerhaftes CDN kann auch die SharePoint Framework-Lösungen beeinträchtigen, und im schlimmsten Fall kann ein beschädigtes CDN dazu führen, dass die Daten im SharePoint (Online)-Mandanten ebenfalls gefährdet sind.

Bei der Genehmigung von SharePoint Framework-Lösungen von Drittanbietern besteht ein übliches Checklistenelement in der Überprüfung der Autorität und Vertrauenswürdigkeit des CDN-Speicherorts sowie aller Drittanbieter, die sie hosten können. Denn nachdem die Anwendung installiert ist und in SharePoint-Websitesammlungen verwendet wird, weisen diese Websitesammlungen eine Abhängigkeit von dem CDN-Speicherort auf. Derzeit gibt es keine einfache Möglichkeit, diesen Endpunkt zu kontrollieren. Der Drittanbieter des CDN kann ohne Wissen des Benutzers mit erwünschten ebenso wie mit unerwünschten Änderungen aktualisieren und so eine Angriffsfläche öffnen, da SharePoint Framework im Kontext des Benutzers ausgeführt wird und alle Aktionen ausführen kann, die dem Benutzer erlaubt sind.

IT-Administratoren wird empfohlen, nachzuverfolgen, welche CDNs verwendet und welche CDNs von der Organisation genehmigt werden – was auch den Enterprise-Entwicklern mitgeteilt werden sollte.

### <a name="office-365-public-cdn"></a>Öffentliches Office 365 CDN

[Öffentliches Office 365 CDN](https://dev.office.com/blogs/general-availability-of-office-365-cdn) ist eine neue Funktion in Office 365 und SharePoint Online, die es Administratoren ermöglicht, statische Elemente, z. B. JavaScript-Dateien, Bilder und CSS-Formatvorlagen, automatisch in einem CDN zu hosten, um bessere Leistung zu ermöglichen. Office 365 Public CDN ist ein geografisch verteiltes Cachefeature, das statische Ressourcen in der Nähe der Endbenutzerbrowser speichert, die diese anfordern.

Administratoren können die Office 365 Public CDN-Funktion in designierten Dokumentbibliotheken aktivieren, die als Ursprung für die statischen Ressourcen dienen sollen. Die Verwaltung der Bibliotheken und des CDN erfolgt mit SharePoint Online PowerShell-Cmdlets. Die Ressourcen in der Dokumentbibliothek werden in das Office 365 CDN repliziert und sind dann über die Office 365 Public CDN-URLs zugänglich, die generiert und der Dokumentbibliothek zugeordnet werden. Alle Updates für die Ressourcen werden innerhalb von 15 Minuten auf die CDN-Endpunkte übertragen. Beachten Sie, dass alle Ressourcen in den Dokumentbibliotheken über den CDN-Endpunkt für anonyme Benutzer zur Verfügung stehen.

## <a name="sharepoint-framework-in-the-enterprise"></a>SharePoint-Framework im Unternehmen

SharePoint war und ist eine der erfolgreichsten Plattformen für die Zusammenarbeit im Unternehmen, und einer der Gründe für den Erfolg war die Möglichkeit, SharePoint zu erweitern und als Plattform für Anwendungen und Integrationen zu betrachten. SharePoint Framework wird diesen Erfolg fortsetzen, indem es SharePoint zu einer moderneren Plattform für unterstützte und standardisierte clientseitige Anpassungen macht.

### <a name="enterprise-developers"></a>Enterprise-Entwickler

SharePoint Framework ermöglicht **Enterprise-Entwicklern**, in der Regel Entwickler, die Anwendungen für die Verwendung in einer Organisation erstellen, SharePoint (Online) in einer strukturierten und unterstützten Weise mit neuen Funktionen zu erweitern. SharePoint Framework bietet alles vom Entwicklungsframework über die Buildpipeline bis zur tatsächlichen Bereitstellung und ermöglicht es Entwicklern, in kurzer Zeit alle Websitesammlungen mit neuen Lösungen und Features zu erweitern, und das alles gesteuert durch den App-Katalog. In einem Unternehmensszenario haben Sie zudem volle Kontrolle über die CDN-Speicherorte – externe oder interne in SharePoint, und Sie können sehr leicht Fixes und Updates in der gesamten Organisation bereitstellen.

Innerhalb des Unternehmens sollten Administratoren und Entwickler gemeinsam eine *Blaupause* für die Bereitstellung von SharePoint Framework-Lösungen erstellen. Die Blaupause sollte Details zu bevorzugten clientseitigen Frameworks, CDN-Speicherorten usw. enthalten. Weitere Informationen finden Sie weiter unten im Abschnitt *Erstellen eines Plans für SharePoint Framework-Anpassungen*.

### <a name="citizen-developers"></a>Einzelentwickler

Einzelentwickler verwenden SharePoint schon seit langem, um mit einer Menge unterschiedlicher Methoden und Techniken Geschäftsanwendungen zu erstellen.
SharePoint Framework wird für bestimmte Szenarien, insbesondere JavaScript-Einbettungen und Lösungen mit Skript-Editor-Webparts, einen großen Fortschritt bedeuten. Es wird diese Lösungen mehr standardisiert und über einen längeren Zeitraum besser verwaltbar machen. Für Einzelentwickler bedeutet es möglicherweise einen gewissen Einarbeitungsaufwand, sich an die neue strukturierte Art der Erstellung von Lösungen zu gewöhnen, die sich jedoch langfristig als stabiler, sicherer und besser verwaltbar erweisen wird.

Wenn die oben genannten Steuerungsmethode *Benutzerdefiniertes Skript* aktiviert ist, dürfen Einzelentwickler keinen beliebigen JavaScript-Code und keine Skript-Editor-Webparts hinzufügen. Dies kann Ihre SharePoint-Umgebung potenziell stabiler und besser verwaltbar machen, aber gleichzeitig bestimmte *Innovationen* in Ihrem Unternehmen verhindern. Und es sollte dafür sorgen, dass sich Ihre Einzelentwickler mit den Enterprise-Entwicklern über die zukünftige Verwendung von SharePoint-Framework einigen.

### <a name="user-experience-designers-and-front-end-developers"></a>Benutzeroberflächendesigner und Front-End-Entwickler

Für Webentwickler oder Benutzeroberflächendesigner wird SharePoint-Framework sehr nützlich sein. Die Workbench ermöglicht Front-End-Entwicklern die Verwendung einer SharePoint Framework-Lösung unter jedem beliebigen Betriebssystem und die Verwendung ihrer bevorzugten Bearbeitungstools ohne SharePoint. Dabei können sie simulierte Daten verwenden und sich auf die Benutzeroberfläche konzentrieren. SharePoint-Framework wird parallel mit [Office UI Fabric](https://dev.office.com/fabric), dem offiziellen Front-End-Entwicklungsframework für Office und Office 365, veröffentlicht und ermöglicht Benutzeroberflächendesignern, eine nahtlose Oberfläche für Office, Office 365 und die integrierten benutzerdefinierten Lösungen zu erstellen.

### <a name="system-integrators-si"></a>Systemintegratoren (SI)

Wenn Sie Systemintegratoren (SI) oder Beratungsfirmen zum Erstellen von SharePoint- und Office 365-Lösungen hinzuziehen, sollten Sie Empfehlungen oder sogar Anforderungen für die Erstellung von SharePoint-Framework-Lösungen stellen, damit sie mit dem Enterprise-Plan für SharePoint-Framework übereinstimmen. Systemintegratoren haben in der Regel eine bevorzugte Art, Lösungen zu erstellen, die nicht immer mit der Governance Ihres Unternehmens konform ist. Daher ist diese Diskussion mit den Systemintegratoren wichtig und macht es letztendlich für alle Parteien leichter.
Ein typisches Szenario mit Systemintegratoren besteht darin, dass diese die Lösung für Ihr Unternehmen erstellen und Sie nach Abschluss des Projekts die Aufgabe haben, die Lösung zu verwalten, upzugraden und zu aktualisieren. Daher ist es umso wichtiger, dass Sie sich mit den SI einigen, wie die SharePoint Framework-Lösungen erstellt und gehostet werden.

### <a name="independent-software-vendors-isv"></a>Unabhängige Softwarehersteller (ISV)

Unabhängige Softwarehersteller (ISV) sind Organisationen, die Drittanbieterlösungen für den Massenmarkt erstellen, und sie erfüllen möglicherweise nicht immer Ihren Plan für SharePoint Framework-Lösungen. Zudem sind ISVs in der Regel Eigentümer ihres Codes und ihres geistigen Eigentums, wodurch es schwierig für Sie ist, ihre Art der Implementierung und des Hostings ihrer Lösungen zu ändern. Bei Verwendung von SharePoint Framework-Lösungen von Drittanbietern müssen Sie genau untersuchen, wie sie Updates verwalten und wie ihre Lösungen gehostet werden. Lassen Sie z. B. zu, dass die Lösung ohne Ihr Wissen aktualisiert wird? Lassen Sie zu, dass die statischen Ressourcen ohne Ihre Kontrolle im CDN des ISV gehostet werden? Wie ist Ihre Vertrauensstellung zu diesem ISV? Bedenken Sie, dass clientseitiger Code in SharePoint Framework im Kontext des aktuellen Benutzers ausgeführt wird und Sie keine Möglichkeit haben, die Ausführung einzuschränken, was mit SharePoint-Add-Ins möglich ist.

## <a name="building-a-plan-around-sharepoint-framework-customizations"></a>Erstellen eines Plans für SharePoint-Framework-Anpassungen

Wenn Sie SharePoint Framework als eines der Tools für die Erweiterung Ihrer SharePoint (Online)-Instanzen einführen möchten, müssen Sie dies planen. Der Plan sollte mit der Einführung des neuen Technologiestapels beginnen, der beim Erstellen von SharePoint Framework-Lösungen verwendet wird. Entwickler benötigen möglicherweise Training zur Verwendung von TypeScript als primäre Sprache zum Schreiben von SharePoint Framework-Code – je nachdem, wo die Entwickler herkommen, z. B. aus der clientseitigen Welt mit JavaScript oder von einem serverorientierten Programmierstil mit C#/.NET, PHP usw. Ein anderer Aspekt von SharePoint Framework, den Entwickler lernen müssen, ist die tatsächliche Toolkette für SharePoint Framework mit node.js, npm und Gulp und wie sie die verschiedenen Gulp-Aufgaben zum Erstellen, Packen, und Bereitstellen von Lösungen verwenden. Eine sehr gute Ausgangsressource hierfür sind die [offizielle SharePoint Framework-Dokumentation](./sharepoint-framework-overview.md) und die [SharePoint Github-Repositorys](https://github.com/SharePoint/).

Möglicherweise möchten sich Entwickler für die Organisation auf ein bestimmtes clientseitiges Framework oder auch auf verschiedene Frameworks festlegen. Zu den clientseitigen Frameworks gehören unter anderem React, Knockout, Angular, Handlebars, jQuery usw. Die Standardisierung auf ein Framework hat Vorteile, da sie Entwicklern ermöglicht, mehr wiederverwendbaren Code zu erstellen und bessere Konsistenz bei der Erstellung und Verwaltung von Lösungen zu erreichen. Andererseits bietet es Vorteile, mehrere Frameworks zuzulassen, da jedes clientseitige Framework seine Vor- und Nachteile bzw. Anwendungsfälle hat. Dagegen kann die Zulassung beliebiger clientseitiger Frameworks zu Fragmentierung in Ihren Enterprise-Lösungen führen. Zudem kann die Verwendung mehrerer unterschiedlicher Frameworks die Seitenladezeit verlängern, da viele Frameworks das Laden mehrerer externer Bibliotheken erfordert.

Standardmäßig bietet der SharePoint Framework Yeoman-Generator Vorlagen für zwei clientseitige Frameworks: React und Knockout. Mit der Zeit ist zu erwarten, dass die Community weitere Generatoren oder Sub-Generatoren hinzufügen wird, um andere clientseitige Frameworks verwenden zu können. Die Wahl von React als Ihr bevorzugtes clientseitiges Framework hat den Vorteil, dass Microsoft eine React-Version der Office UI Fabric erstellt hat. Somit erhalten Sie das Aussehen und Verhalten von Office und Office 365 für Ihre Anpassung, falls dies in Ihrer Organisation bevorzugt wird.

Der vierte Aspekt der Planung ist, wie und wo Sie Ihre Lösungsartefakte bereitstellen, d. h. in welchem CDN die generierten Skriptbündel und -objekte gespeichert werden. Standardmäßig werden in den Gulp-Aufgaben in der Toolkette nur Azure Blob Storage und Azure CDN unterstützt. Dies ist möglicherweise gut geeignet, wenn Sie ein Azure-Abonnement verwalten können und Ihre Ressourcen für mehrere Mandanten freigeben müssen. Ein weiteres sehr gängiges Szenario ist die Verwendung von SharePoint Online und seinem CDN-Feature als Host für die Artefakte. Dafür müssen Sie jedoch die Toolkette ändern und optional benutzerdefinierte Gulp-Aufgaben verwalten.

Abschließend müssen Entwickler über Application Lifecycle Management (ALM) nachdenken. Dies betrifft die Art, wie Sie Quellcode und Versionsverwaltung, automatische Erstellung, Tests, Bereitstellung usw. verwalten. Am häufigsten werden Systeme zur Quellcode-Versionsverwaltung verwendet, wie z. B. Git, Github oder Visual Studio Team Systems. Für Continuous Integration gibt es keine Standardtools. Sie können Ihr bevorzugtes Tool verwenden, das node.js unterstützt, z. B. Visual Studio Team Systems, Travis CI oder Jenkins. Mit diesen Tools können Sie den Build- und Testprozess automatisieren und im Falle eines erfolgreichen und genehmigten Builds sogar automatisch die Artefakte im CDN-Speicherort bereitstellen. So können Sie alles von der Entwicklerprüfung im Code über die Bereitstellung bis zur Produktion automatisieren.

## <a name="management-capabilities-of--sharepoint-framework-solutions"></a>Verwaltungsfunktionen von SharePoint Framework-Lösungen

Alle SharePoint Framework-Lösungen, die in einem Mandanten bereitgestellt werden, müssen von einem Mandantenadministrator genehmigt werden. Dies erfolgt durch Hochladen des SharePoint Framework-Pakets, der `.sppkg`-Datei, in die *Apps für SharePoint*-Bibliothek. Wenn eine neue Lösung zur Bibliothek hinzugefügt wird, wird der Administrator in einem Dialogfeld aufgefordert, die Lösung für die Verwendung im Mandanten zu genehmigen. Im Dialogfeld wird erläutert, dass es sich um eine Lösung mit *voll vertrauenswürdigem clientseitigem Code* ohne Ressourceneinschränkungen handelt und dass sie im Kontext des Benutzers ausgeführt wird. In dem Dialogfeld ist auch angegeben, aus welcher Domäne hauptsächlich Inhalte abgerufen werden, d. h. der CDN-Speicherort der SharePoint Framework-Skripts. Beachten Sie, dass jede SharePoint Framework-Anwendung nach dem ersten Laden aus dem CDN auch Daten von anderen Speicherorten laden kann. Einmal genehmigt, kann die SharePoint Framework-Lösung in jeder Websitesammlung aktiviert werden.

![SharePoint-Framework-Dialogfeld zu Vertrauensstellung für App-Katalog](../images/enterprise-guidance-trust-spfx-dialog.png)

Ein Administrator des App-Katalogs kann das Paket jederzeit aus dem App-Katalog entfernen, indem er das Lösungspaket aus der *Apps für SharePoint*-Bibliothek entfernt. Dadurch wird die Verwendung der Lösung in allen Websitesammlungen verhindert. Die Lösung kann auch durch Ändern der *Enabled*-Eigenschaft des hochgeladenen Pakets deaktiviert werden. Dadurch wird die Lösung sofort in allen Websitesammlungen deaktiviert. Auf vorhandenen Seiten, die clientseitige Webparts verwenden, wird das Webpart nicht gerendert, und die App ist nicht mehr in den Websitesammlungen verfügbar bzw. kann in vorhandenen Websitesammlungen nicht mehr hinzugefügt werden. Wenn Sie eine SharePoint-Framework-Lösung entfernen, werden keine Daten oder Informationen entfernt, die durch die eigentliche clientseitige Lösung in SharePoint oder in einer von der Lösung verwendeten externen Datenquelle erstellt wurden.

Der Administrator kann auch andere Eigenschaften für das Paket im App-Katalog ändern, um die *Sichtbarkeit* der Lösung in Websitesammlungen zu verbessern, z. B. das Symbol, die Kategorie, die Beschreibung und den Status.

Wenn das Lösungspaket aktualisiert werden muss, z. B. wenn neue SharePoint Framework-Artefakte erstellt oder andere Änderungen auf Paketebene vorgenommen werden, muss der Administrator nur eine neue Version des Pakets in die Bibliothek hochladen.

Wie bei SharePoint-Add-Ins kann ein Mandantenadministrator auch die SharePoint Framework-Lösungen überwachen. Im SharePoint Admin Center unter *apps* kann der SharePoint-Administrator die SharePoint Framework-Lösungen hinzufügen und dann sehen, wie viele installierte Instanzen es von einer bestimmten Lösung gibt. Dies gilt für SharePoint-Add-Ins ebenso wie für SharePoint Framework-Lösungen.

Um eine SharePoint Framework-Lösung in einer Websitesammlung zu aktivieren, muss sie der Websitesammlungsadministrator zu der Websitesammlung hinzufügen. Dies geschieht wie bei SharePoint-Add-Ins, indem Sie in der Websitesammlung *Neue App hinzufügen* auswählen und dann die Lösung aus der Liste der Apps auswählen. Sobald die App hinzugefügt wurde, steht sie für die Verwendung in der Websitesammlung zur Verfügung. Der Websitesammlungsadministrator kann die SharePoint Framework-Lösung auch aus der Websitesammlung entfernen. Hierzu wechseln Sie zu *Websiteinhalte* und wählen dann *Entfernen* für die App aus.

### <a name="sharepoint-framework-deployment-scopes"></a>SharePoint-Framework-Bereitstellungsbereiche

Beim Erstellen von SharePoint-Framework-Lösungen können Entwickler entscheiden, ob ihre Lösung die mandantenweite Bereitstellung unterstützt oder ob sie für jede Website separat bereitgestellt werden muss. Letzteres ist erforderlich, wenn die Lösung zusätzliche Ressourcen wie Listen bereitstellen muss, nachdem sie für eine Website bereitgestellt wurde.

Während Entwickler beim Erstellen der Lösung entscheiden, ob die Lösung die mandantenweite Bereitstellung unterstützt, treffen Administratoren die endgültige Entscheidung darüber, wie die Lösung bereitgestellt wird. Auch wenn Sie eine Lösung für alle Websites im Mandanten bereitgestellt werden könnte, können Administratoren festlegen, dass sie nur für bestimmte Websites bereitgestellt wird. Wenn die Lösung die mandantenweite Bereitstellung nicht unterstützt, können Administratoren sie nur für bestimmte Websites bereitstellen.

> **Wichtig:** Die mandantenweite Bereitstellung ist nur in SharePoint Online verfügbar. In lokalen SharePoint-Umgebungen können SharePoint-Framework-Lösungen nur für bestimmte Websites bereitgestellt werden können.

SharePoint Framework verfügt im Gegensatz zu SharePoint-Add-Ins nicht über einen Store. Aus diesem Grund müssen Bereitstellungen immer durch den Mandantenadministrator durch Hinzufügen und Genehmigen eines Lösungspakets im App-Katalog initiiert werden.

### <a name="backing-up-and-restoring-sharepoint-framework-components"></a>Sichern und Wiederherstellen von SharePoint Framework-Komponenten

SharePoint Framework-Lösungen haben keinen bestimmten Sicherungs- und Wiederherstellungsfeatures. Die einzige Empfehlung aus administrativer Sicht lautet, eine Kopie aller installierten Lösungspaketdateien (`.sppkg`) aufzubewahren, für den Fall, dass ein Lösungspaket versehentlich aus dem App-Katalog entfernt wird. Der App-Katalog ist jedoch eine SharePoint-Bibliothek und umfasst dieselben Funktionen wie jede Dokumentbibliothek, mit aktivierter Hauptversionsverwaltung und Papierkorb.

Was Sie nicht sichern können, sind die tatsächlichen Lösungsartefakte wie z. B. Skriptbündel und Ressourcen, die in einem CDN gehostet werden. Wenn Sie jedoch die Kontrolle über das CDN haben oder das CDN eine SharePoint-Website ist, können Sie es sichern. Wenn Sie dagegen SharePoint Framework-Lösungen von einem Drittanbieter verwenden, gibt es unter Umständen keine Möglichkeit für Ihre Organisation, sie zu sichern.

## <a name="sharepoint-framework-roadmap"></a>SharePoint Framework-Roadmap

SharePoint-Framework hat seit Februar 2017 den Status der allgemeinen Verfügbarkeit (General Availability, GA). „Allgemeine Verfügbarkeit“ (GA) bedeutet, dass die IT und Entwickler SharePoint-Framework mit entsprechendem Support in der Produktion verwenden können. Über die allgemeine Verfügbarkeit hinaus erwarten wir, dass die Szenarien für die Erstellung und Verwendung von SharePoint Framework-basierten Komponenten über Webparts-Szenarien hinaus auf Bereiche wie Listen- und Websiteanpassungen erweitert werden können. Weitere Informationen zu SharePoint-Framework finden Sie im dedizierten [SharePoint-Framework-Roadmap-Artikel](./roadmap.md).

Wesentliche Änderungen oder die Einführung neuer wichtiger Features werden über das Office 365-Nachrichtencenter angekündigt, das Sie in Ihrem Mandantenadministrator finden – etwas, das ein Office 365-Administrator im Rahmen seiner täglichen Routineaufgaben prüfen sollte. Eine weitere wichtige Ressource ist der [Office Developer-Blog](http://dev.office.com/blogs). Hier finden Sie weitere Details und Updates.

## <a name="summary"></a>Zusammenfassung

SharePoint-Framework ist eine hervorragende Ergänzung und Weiterentwicklung der Toolbox für SharePoint-Anpassungen, mit der Entwickler auf unterstützte und kontrollierbare Weise SharePoint erweitern können. Mit SharePoint-Framework, das auf quelloffenen und modernen Technologien basiert, können Entwickler Ihre SharePoint Enterprise-Entwicklung nicht nur für das SharePoint-Team, sondern auf eine Reihe anderer Entwickler erweitern.  Durch Bereitstellen der richtigen Governance und Unterstützung für SharePoint-Framework in Ihrem Mandanten helfen Sie als Administrator den Entwicklern, schneller ansprechendere Lösungen zu erstellen, was zu rund um besserer Effizienz führt.

Da SharePoint Framework für Entwickler von Erst- und Drittanbietern erstellt wurde und von Microsoft zunehmend für zukünftige Featureerweiterungen von SharePoint verwendet wird, stellt es auch für Ihre Organisation eine gute Wahl dar. Wir erwarten in Zukunft weitere inkrementelle Updates und Ergänzungen von SharePoint Framework, um die Funktionslücke zwischen dem klassischen SharePoint und der modernen SharePoint-Umgebung zu schließen.
